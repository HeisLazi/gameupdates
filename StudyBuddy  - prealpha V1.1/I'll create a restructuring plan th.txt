KEY PRINCIPLES
==============

1. TESTER ENTRY POINT:
   - Open START_HERE.md → tells them to run build.sh → open build/StudyBuddy.html
   - Everything else is "don't touch unless debugging"

2. AI/DEVELOPER ENTRY POINT:
   - Read CONTEXT.md (high-level system map)
   - Read docs/ARCHITECTURE.md (how systems connect)
   - Files in src/ are named by *feature*, not build order
   - Comment at top of each file shows what it does + dependencies
   - Clear section headers: [CORE] = required, [OPT] = optional, [SYSTEM] = runtime

3. BUILD FILE GROUPING:
   - Not alphabetical or build-order
   - Grouped by: Core → Study → Arena → Modes → Systems
   - Makes it obvious what a new feature should touch

4. DOCUMENTATION STRATEGY:
   - START_HERE.md = 1 min read for testers (just run the game)
   - README.md = project context (what is this, who made it)
   - CONTEXT.md = AI/dev quick reference (what file does what)
   - docs/ = deep dives (architecture, schema, design decisions)
   - Each file in src/ has 2-line header comment explaining purpose

5. DESIGN SEPARATION:
   - design/ folder is READ-ONLY for developers
   - tokens.css is the source of truth (don't hardcode colors)
   - High-fidelity mockups exist but aren't spliced—use as reference
   - Clear file headers: "Design reference only — port markup to src/"

6. DATA HANDLING:
   - modules_all.json, deadlines.json, grades.json live in src/
   - Edited via the app itself (not hand-edited in most cases)
   - Clear schema in docs/MODULE_SCHEMA.md
   - JSON files are auto-preserved across builds


FILE NAMING CONVENTION
======================

Core files (always needed):
  ✓ app.jsx, head.html, arena.css

Screen/mode files (named by purpose, not function):
  ✓ splash.jsx, library.jsx, combat.jsx, skills.jsx
  ✓ NOT: "arena_views_combat_hud.jsx" → just combat.jsx

System files (named by domain):
  ✓ bonuses.jsx (prestige math), raids.jsx (raid system)
  ✓ NOT: "helpers.jsx", "utils.jsx" → be specific

New features:
  → Create cram.jsx, not cram_mode.jsx or cram_handler.jsx
  → Put in src/ root or a feature folder, file header explains everything


BUILD PROCESS (SIMPLIFIED)
==========================

For testers:
  1. Double-click build.sh (or build.bat on Windows)
  2. Game opens automatically
  3. Done.

For devs:
  1. Edit src/*.jsx
  2. cd src && python3 build_html.py
  3. Hard refresh build/StudyBuddy.html
  4. Verify in browser

For AIs:
  1. Read CONTEXT.md to understand system
  2. Read docs/ARCHITECTURE.md to see how systems connect
  3. Search src/ for the feature you need to edit
  4. File headers explain dependencies + key functions
  5. Check docs/API.md for state shape if unsure
